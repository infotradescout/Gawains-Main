# RoundTable Discord Packet + Authority Contract

RoundTable owns the Discord packet doctrine, terminal records, and authority contract.
Merlin owns Discord webhook dispatch, Discord interaction signature verification, and verified approval record writing.

This document is a contract. It is not runtime code and does not configure Discord.

## Ownership Boundary

RoundTable owns:

- Canonical Discord packet shape.
- Delivery eligibility doctrine.
- Terminal packet records.
- Authority doctrine.
- The rule that text fields are never approval authority.
- Evidence expectations that Merlin must return after delivery or approval verification attempts.

Merlin owns:

- Discord webhook dispatch.
- Discord interaction signature verification.
- Allowlist, guild, channel, role, packet ID, and action-scope verification.
- Verified approval record persistence in Merlin runtime storage.
- Safe failure handling for dispatch and interaction verification.

RoundTable must not contain:

- Discord webhook secrets.
- Discord bot runtime.
- Discord API dispatch code.
- Signature verification implementation.
- Live Discord send behavior.
- Auto-merge, auto-deploy, auto-send, auto-apply, Drive mutation, or product mutation behavior.

## Canonical Discord Packet Shape

A canonical RoundTable Discord packet MUST include:

```text
packetId
packetType
createdAt
sourceRepo
sourceBranch
sourceCommit
lane
title
summary
requestedAction
approvedActionScopes
authorityRequired
delivery
evidence
doctrineRefs
terminalRecordRequired
```

The packet MAY include:

```text
riskLevel
expiresAt
relatedPackets
humanRecipients
markdownBody
```

Required field meanings:

- `packetId`: stable unique packet ID generated by RoundTable.
- `packetType`: must identify the packet as a Discord delivery packet or approval-request packet.
- `createdAt`: ISO timestamp from RoundTable at packet creation.
- `sourceRepo`: repo that produced the packet or requested delivery.
- `sourceBranch`: branch associated with the request.
- `sourceCommit`: commit SHA or `OPEN QUESTION` when not yet known.
- `lane`: RoundTable lane name.
- `title`: short human-readable title.
- `summary`: bounded summary of the decision, review, or request.
- `requestedAction`: exact action being requested from humans or from the execution layer.
- `approvedActionScopes`: exact scopes that can be approved by Discord interaction.
- `authorityRequired`: human or council authority required before execution.
- `delivery`: delivery target metadata that contains no secrets.
- `evidence`: references to source packets, review packets, validation output, or merge decisions.
- `doctrineRefs`: RoundTable or Albion doctrine documents that govern the packet.
- `terminalRecordRequired`: whether RoundTable must receive and store a terminal evidence packet.

## Delivery Eligibility

A packet is eligible for Merlin Discord delivery only when all are true:

- The packet has a stable `packetId`.
- The packet has an explicit `requestedAction`.
- The packet has explicit `approvedActionScopes`.
- The packet has a non-secret delivery target reference.
- The packet states the required human or council authority.
- The packet states that Discord delivery is evidence only until Merlin writes a verified approval record.
- The packet does not ask Merlin to execute, merge, deploy, send, apply, mutate Drive, or mutate product state.
- The packet is recorded or recordable by RoundTable as a terminal packet.

A packet is not eligible for delivery when:

- It depends on `approvalStatus`, `approvedBy`, or any other text field as authority.
- It contains webhook URLs, tokens, public-key secrets, bot tokens, or private runtime credentials.
- It asks RoundTable to run runtime dispatch or signature verification.
- It asks Merlin to treat message delivery, a reaction, or plain text as approval.
- It lacks the action scope that the approval would authorize.

## Verified Approval Record Requirements

A verified approval record is valid only when Merlin has verified all required runtime facts:

```text
packetId
actionScope
discordInteractionId
discordUserId
discordGuildId
discordChannelId
discordRoleIds
signatureVerified
allowlistedUserVerified
guildVerified
channelVerified
roleVerified
packetIdVerified
actionScopeVerified
verifiedAt
verificationSource
noExecutionPerformed
```

Merlin must write the verified approval record only after:

- Discord interaction signature verification passes.
- The Discord user ID is allowlisted.
- Required guild, channel, and role checks pass.
- The packet ID exists in Merlin runtime storage as a RoundTable-origin packet.
- The requested action scope matches one of the packet's `approvedActionScopes`.

The record must state `noExecutionPerformed: true` unless a separate, explicitly authorized execution lane performs the action after approval verification.

## Non-Authority Fields

`approvalStatus`, `approvedBy`, message text, Markdown text, button labels, summaries, comments, and free-form human-readable fields are never authority.

They may be displayed, indexed, or recorded as context. They must not be used to approve a merge, deploy, send, apply, Drive mutation, product mutation, onboarding action, public statement, vendor contact, financial action, or authority-sensitive change.

## Discord Reactions

Discord reactions are not approval.

A reaction may become evidence only if Merlin converts the interaction into a verified approval record that satisfies this contract. Until then, it is a visible reaction, not authority.

## Terminal Evidence Back To RoundTable

After Merlin attempts delivery or approval verification, Merlin should return an evidence packet to RoundTable containing:

- `packetId`
- delivery attempted or not attempted
- delivery status
- failure reason, if any
- Discord message ID, if available
- verified approval record ID, if created
- verification checks performed
- `noExecutionPerformed`

RoundTable records the terminal packet. RoundTable does not infer approval from incomplete evidence.

## Prohibited Outcomes

This contract does not authorize:

- Live Discord send by RoundTable.
- Runtime Discord code in RoundTable.
- Webhook secret storage in RoundTable.
- AI-generated approval authority.
- Approval by `approvalStatus` or `approvedBy` text.
- Approval by raw Discord reaction.
- Auto-merge.
- Auto-deploy.
- Auto-send.
- Auto-apply.
- Drive mutation.
- Product mutation.
